### Madre — Diagnóstico de unhealthy

```bash
docker inspect --format='{{.State.Health.Status}}' open-webui qdrant local_tripwire
# → unhealthy / unhealthy / starting

docker inspect open-webui --format='{{json .State.Health}}' | jq
docker inspect qdrant --format='{{json .State.Health}}' | jq
```

**Causa raíz identificada:** el healthcheck definido en el compose ejecuta `wget`,
pero ese binario no existe dentro de los contenedores `open-webui` ni `qdrant`.

```
"Output": "/bin/sh: 1: wget: not found\n"
"FailingStreak": 390
```

**Los servicios funcionan**: los logs de `open-webui` muestran peticiones HTTP 200
continuas desde `100.86.119.102` y Qdrant arranca correctamente en puertos 6333/6334.
El `unhealthy` es un **falso negativo del healthcheck**, no un fallo de la aplicación.

**Pendiente (nuevo issue):** corregir el healthcheck sustituyendo `wget` por `curl` o
by el endpoint nativo de la imagen:

```yaml
# open-webui — fix propuesto
healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
  interval: 30s
  timeout: 10s
  retries: 3

# qdrant — fix propuesto
healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:6333/healthz"]
  interval: 30s
  timeout: 10s
  retries: 3
```

## Investigación externa — Auditoría Gemini

Se recibió una auditoría técnica externa (vía Gemini) sobre el ecosistema Yggdrasil-DEW.
Se documenta aquí como investigación de referencia, **no como estado ejecutado**.

### Hallazgos relevantes

**Estructurales:**
- SPOF en el `.env` de Madre: las variables de entorno se tratan como configuración
  estática, deberían gestionarse como secretos (SOPS/Vault).
- Docker-compose drift: la existencia de "versión 16" de archivos indica cambios
  manuales fuera del flujo GitOps. El repo debe ser la única fuente de verdad.

**Documentación:**
- Falta `docs/secrets.md`: ciclo de vida de tokens (rotación sin exponer a GitHub).
- Falta runbook `docs/runbooks/DRP.md`: Plan de Recuperación ante Desastres (RTO < 2h).

**Operabilidad:**
- Falta healthcheck en contenedores críticos (detectado hoy de forma práctica ✓).
- Falta capa de agregación de logs (Promtail + Loki) para correlación centralizada.
- No existe DRP documentado: tiempo de recuperación del RAG/VectorDB sin definir.

**Risk Assessment adicional:**
- HDD con 28k horas (issue #31): implementar `btrfs scrub` semanal automatizado.
- GitHub Secret Scanning: verificar `Settings → Security → Secret scanning` por
  leaks históricos en historial de commits.

### Propuestas técnicas recibidas

1. `scripts/env-checker.sh` — valida `.env` contra `.env.template` antes de deploy.
2. `docs/runbooks/DRP.md` — procedimiento de reconstrucción del stack desde cero.
3. Alias `ygg-check` en `.bashrc`/`.zshrc` para validación rápida pre-commit.
4. GitHub Actions workflow para validar `.env` en cada `git push`.

> Estas propuestas quedan como **backlog** hasta verificar implementación real.
> Ver [[MASTER-PENDIENTES]] para priorización.
