## 🔴 Incidente de seguridad — Exposición de secretos en chat

**Hora aproximada:** 15:00 CEST

Durante la sesión de rotación del token de Telegram de THDORA, se ejecutó `cat -A .env | head -10` y el output completo del `.env` fue pegado en el chat de Perplexity, exponiendo:

| Credencial | Acción tomada |
|---|---|
| `TELEGRAM_BOT_TOKEN` | Revocado en BotFather → token nuevo generado y aplicado |
| `GROQ_API_KEY` | Revocada en panel Groq |
| `GITHUB_TOKEN` (ghp_...) | Revocado en GitHub Settings → Tokens |
| `ADMIN_TOKEN` | Rotado manualmente |

**Causa raíz:** El `.env` completo fue visible porque el comando se ejecutó sin filtrar el output. El archivo no tenía el bug de espacios que se sospechaba — las claves estaban bien formadas.

**Lección aprendida:** Antes de pegar output de terminal en un chat externo, revisar visualmente que no contenga secretos. Añadir `grep -v TOKEN | grep -v KEY | grep -v SECRET` como filtro de seguridad al revisar `.env`.

## 🔴 Incidente — Pérdida de datos en yggdrasil-secops

**Hora:** ~15:34 CEST

Al ejecutar `rm -rf /home/varopc/repos/yggdrasil-secops` sin revisar el `git status` previo con atención, se perdieron dos directorios que existían como **untracked files** (nunca commiteados a Git):

- `docs/pentesting/`
- `docs/seguridad/`

Verificado que **no existen** en la copia buena (`/home/varopc/yggdrasil-secops/docs/`), por lo que el contenido es irrecuperable.

**Causa raíz:** El `git status` mostró el aviso `Untracked files: docs/pentesting/ docs/seguridad/` pero se procedió al `rm -rf` inmediatamente sin evaluar si ese contenido era valioso.

**Lección aprendida crítica:** Antes de cualquier `rm -rf` sobre un directorio con repo Git, leer el output de `git status` completo. Si hay untracked files, decidir explícitamente: `git add` + commit, mover a otro lugar, o confirmar conscientemente que se puede perder.

**Regla operativa a añadir al playbook:**
```bash
# Antes de rm -rf en cualquier repo
git status
git stash list
# Solo si ambos están limpios → proceder
```
