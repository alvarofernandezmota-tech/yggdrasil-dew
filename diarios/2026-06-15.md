## 🛠️ Sesión tarde — Rescate Redmi A5 (~14:00 CEST)

### Dispositivo
- **Modelo:** Xiaomi Redmi A5 | **Chip:** Unisoc T765 (Qogirl6) | **Serial:** `863d0058304831323851135e23407c`
- **Problema:** Sistema roto + Recovery (ISO) dañado — no se puede entrar a nada

### Cómo se procedió
Diagnóstico sistemático de arriba a abajo:
1. Recovery → dañado (ISO rota)
2. ADB → `unauthorized` (pantalla rota, no se puede aceptar diálogo de depuración)
3. `fastboot oem unlock` → `Err:0xffffffff` (bootloader bloqueado OEM Xiaomi, definitivo)
4. `fastboot flashing unlock` → `unknown cmd` (no soportado)
5. Identificación del chip por el script `.sh` de flash → `qogirl6_pubcp_MHM` confirma Unisoc T765
6. Conclusión: única salida viable es **EDL / SPRD Download Mode**

### Herramientas instaladas en la sesión
```bash
sudo pacman -S android-tools android-udev usbutils
sudo usermod -aG adbusers varopc
```

### Estado final
| Componente | Estado |
|---|---|
| ADB | ❌ `unauthorized` |
| `fastboot oem unlock` | ❌ `Err:0xffffffff` |
| `fastboot flashing unlock` | ❌ `unknown cmd` |
| Modo fastboot | ✅ Funciona |
| Recovery / ISO | ❌ Dañado |
| Sistema operativo | ❌ Roto |

#### Decisiones técnicas clave
- Patrón provisional+edit para evitar TimedOut de Telegram (~5-15s)
- Few-shot examples JSON en system prompt → el modelo aprende el formato exacto
- `asyncio.gather(return_exceptions=True)` → contexto citas+hábitos en paralelo, un fallo no cancela el otro
- Patrón Strategy LLM: `GroqBackend` / `OllamaBackend` con fallback automático en `ConnectError`
- `LLMBackendFactory` configurable por variable de entorno `LLM_BACKEND`
- `calculate_free_slots()` función pura sin I/O → testeable directamente
- `batch_alter_table` Alembic → compatible SQLite + PostgreSQL
- `set -e` en SSH script → cualquier fallo corta el pipeline y dispara alerta Telegram
